    Statement Names

    Statement Names

    The intent of the following taxonomy is to define a standard such that the name for a given statement is:

    1. Constructed from attributes that can be unambiguously parsed from a piece of raw sql

    2. Structured such that user provided names can be easily implemented and loosely parsed into the same set of attributes as those generated from (1)

    Every statement has a Name with a set of filterable properties that are used by the rest of the API; for each filterable property, there is an underlying generated (_ge) and provided (_pr) attribute from which it sources the final value.

    The generated attributes are populated for all statements, whereas only those with a name provided in a tag will have populated provided attributes.

    Note

    The final nm for a statement is a parent property over the top of its nm_pr and nm_ge, the value of which reflects the nm_pr if present and the nm_ge otherwise.

    This resolution order is repeated across the underlying components of nm, documented below.

    Anchor & Description

    A statement name is comprised of the following components, with context on the information provided by each directly below:

    {anchor}{delimiter}{desc}

    anchor
    what operation is a statement performing

    on what kind of object is it operating

    delimiter
    a configured value with which to delimit the anchor and the desc

    desc
    A free-form piece of text associated with the statement


    kw, obj, & desc_ge

    As a Statement is parsed, a Name is constructed based on following the structure:

    {kw} {obj}{delimiter}{prefix} {index}

    kw
    the first sql keyword the statement contains

    obj
    the first identified named object within the first line defined by a generalized DDL standard

    prefix
    a configured value with which to prepend a statement's relative index when generating a desc

    index
    the statement's relative index position within the script

    The components defined above represent a final set of statement-level attributes after has resolved the information parsed from its sql and any markup that is (optionally) included

    snowmobile.Table¶

    snowmobile.Table is a data loading solution that at minimum can be provided a Connector (sn), a pandas.DataFrame (df), and a table name.

    The rest of its keyword arguments are mirrored in the [loading.default-table-kwargs] section of snowmobile.toml and documented in **the API Docs for Table.

    snowmobile.Table

    Note

    Similarly to how Connector handles its creds keyword argument, Table will adhere to any arguments explicitly provided and differ to the values configured in snowmobile.toml otherwise.

    The exact configuration that a given instance of Table will differ to is defined by the state of snowmobile.toml at the time the Connector (sn) was instantiated, not its state at the time that it (Table) was instantiated; also outlined in Connector description.

    The default behavior outlined below reflects those within the [loading] section of the default snowmobile.toml file.

    Core Features¶

    No embedding credentials within scripts

    • import snowmobile is all that's required to connect

    Access multiple sets of connection parameters, credentials, and other configuration options

    • Accessed by user-assigned alias

    Map in-warehouse file formats to stored DDL and save options

    • Enables flexible, controlled loading of data

    Manipulate .sql scripts as Python objects

    • Queue scripts across sessions

    • Name and access individual statements as a dictionary off a Script object

    • Document scripts with a sql-compliant markup syntax that exports as markdown

    • Work with partitions of scripts based on assigned tags and other metadata parsed by snowmobile

    Access a single configuration file, snowmobile.toml, across any number of Python instances on a given machine

    • Location tracking of snowmobile.toml is handled by snowmobile

    • A path to a specific snowmobile.toml file can be specified for long-standing or containerized processes

    Use a single method to return query results in a SnowflakeCursor or a DataFrame

    Built-in methods to run common information schema and administrative commands
